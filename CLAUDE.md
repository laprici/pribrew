# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Pribrew — a coffee-extraction logbook (bitácora de extracción). Multi-user, where recipes/beans are **shared per group**. PWA (web sidebar / mobile bottom-nav, installable). The codebase and UI copy are in **Spanish**; keep new user-facing strings, comments, and commit messages in Spanish to match.

## Commands

```bash
pnpm install          # repo is on pnpm (pnpm-lock.yaml); package-lock.json is stale
pnpm dev              # vite dev server → http://localhost:5173
pnpm build            # tsr generate → tsc --noEmit → vite build (with PWA)
pnpm typecheck        # tsr generate → tsc --noEmit  (the closest thing to a test/CI check)
pnpm preview          # serve the production build
```

There is **no test runner and no linter** configured. `pnpm typecheck` (or `pnpm build`) is the verification gate — run it after changes. Note `tsconfig.json` sets `noUnusedLocals`/`noUnusedParameters` to **false**, so unused vars won't fail the build.

### Local backend (Supabase, requires Docker)

```bash
supabase start        # Postgres, Auth, Storage, Studio
supabase db reset     # applies migrations/0001_init.sql + seed.sql (7 methods)
supabase status       # read API URL + anon key into .env.local
```

Frontend env (`.env.local`, not committed — README references a `.env.example` that is absent):
`VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. For frictionless local auth, set `enable_confirmations = false` under `[auth.email]` in `supabase/config.toml`.

## Architecture

**Stack:** Vite + React 18 + TypeScript, Tailwind, TanStack Router (file-based) + TanStack Query, Zod, Supabase. Path alias `@/*` → `src/*`. Zustand is a dependency but not yet used.

**No backend code of our own** — the app talks directly to Supabase from the browser via the `supabase` client. All authorization lives in Postgres (RLS + RPCs), not in the frontend. This is the single most important thing to understand before changing data access.

### The security model (lives in `supabase/migrations/0001_init.sql`)

Every data table (`beans`, `grinders`, `brews`) follows one pattern enforced by RLS:
- **Read:** own rows OR rows whose owner shares a group with you (`shares_group_with(owner_id)`).
- **Write/update/delete:** own rows only (`owner_id = auth.uid()`).

`owner_id` defaults to `auth.uid()` at the DB level, so inserts don't set it from the client. Group sharing is transitive read access computed by SECURITY DEFINER functions (`is_group_member`, `shares_group_with`) — these are SECURITY DEFINER specifically to avoid RLS recursion on `group_members`.

Operations that can't be expressed as simple RLS policies go through **RPCs** (call with `supabase.rpc(...)`):
- `create_group(group_name)` — inserts group + owner membership atomically (direct group INSERT is not allowed by policy).
- `accept_invitation(invite_token)` — validates token vigencia/usos, adds membership.
- `invitation_preview(invite_token)` — public (anon-grantable) read of group name + validity for the invite landing page.

When adding a feature that needs cross-user access or multi-row atomicity, add an RPC + RLS policy in a **new migration** rather than working around it in the client.

### Frontend layout (`src/`)

- `lib/` — `supabase.ts` (client), `auth.tsx` (`AuthProvider` + `useAuth`, wraps Supabase session), `queryClient.ts` (Query defaults: 30s staleTime, no refetch-on-focus).
- `domain/` — Zod schemas + pure helpers, the source of truth for shapes. `method.ts` holds the 7 `METHOD_KEYS` and labels. `brew.schema.ts` validates per-method params via a **Zod `discriminatedUnion` on `method`**, stored as `brews.method_params` jsonb — DB does not validate this, the app does. `calc.ts` is pure functions (ratio, daysOffRoast, formatTime).
- `data/` — TanStack Query hooks wrapping Supabase calls (e.g. `groups.ts`). New data access goes here as `useX` hooks; invalidate `queryClient` keys on mutation.
- `routes/` — **file-based routing**, auto-generated into `src/routeTree.gen.ts` (do not edit by hand; regenerated by `tsr generate` on build/typecheck). Each route uses `createFileRoute`. `invite.$token.tsx` is the public invite page; dynamic segments use `$param`.
- `components/AppShell.tsx` — the responsive shell (desktop sidebar / mobile bottom-nav) and the **auth gate**: redirects to `/login` when there's no session. Page routes render their content inside `<AppShell title=...>`.

### Design system

Tailwind theme in `tailwind.config.js` defines the palette (`paper`, `surface`, `ink`, `extraction`, etc.) and component classes in `src/styles.css` (`@layer components`): `card`, `btn-primary`/`btn-ghost`, `field`, `label`, and the signature **`readout`/`readout-label`** (mono, tabular-nums) used for numeric coffee readouts (ratio, °C, time, days-off-roast). Reuse these classes instead of ad-hoc styling.
